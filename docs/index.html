Getting started

inkboom is an MCP server that gives your AI agent a persistent design system — validated tokens, component definitions, and resolved token bindings — so that every page it generates is visually consistent.

Prerequisites: Node 22+, pnpm. The server runs on localhost:4778/mcp using the MCP Streamable HTTP transport.

Installation

Clone the repository and install dependencies:

terminal 
❯ git clone https://github.com/rhighs-lab/inkboom
❯ cd inkboom
❯ pnpm install

Start the MCP server:

terminal 
❯ pnpm mcp
{"level":"info","port":4778,"msg":"inkboom MCP HTTP server started"}

The server runs at http://localhost:4778/mcp. Projects are stored at ~/.inkboom/projects/<slug>/project.json.

MCP clients

Claude Code (recommended)

Add to your claude_desktop_config.json or open Claude Code and add the server via settings:

claude_desktop_config.json 
{
  "mcpServers": {
    "inkboom": {
      "command": "npx",
      "args": [
        "tsx",
        "/path/to/inkboom/src/mcp/http-server.ts"
      ]
    }
  }
}

Cursor / other MCP-compatible clients

First start the server with pnpm mcp, then point your client at http://localhost:4778/mcp. The server uses the MCP Streamable HTTP transport (POST to /mcp).

SKILL.md

inkboom ships a SKILL.md file designed to be loaded as a Claude Code skill. It instructs the agent on the designer workflow, token rules, component guidelines, and icon rendering strategy.

Place it in your skills directory or reference it from your CLAUDE.md:

CLAUDE.md 
# Load the inkboom design skill
Read /path/to/inkboom/SKILL.md before any design or UI work.

The SKILL.md covers:

  • The correct tool call order (create_project → set_tokens → set_design_system)
  • Token constraints: 6-digit hex, unit-suffixed dimensions, WCAG AA contrast
  • Component requirements: which 20 base components must always exist
  • Icon rendering: inline SVG from tokens.icons, never emoji
  • Breakpoint usage from tokens.layout

First project

Once the server is running and connected to your agent, the typical workflow is:

agent conversation (abbreviated) 
// Step 1 — create an empty project
create_project({
  slug: "my-site",
  brief: "SaaS landing. Minimal, light, indigo accent."
})
// → { projectId: "my-site", createdAt: "..." }

// Step 2 — set design tokens
set_tokens({
  projectId: "my-site",
  tokens: {
    "$schema": "https://design-tokens.org/schema.json",
    "_meta": { "version": "1.0.0", "generatedAt": "2026-01-01T00:00:00Z" },
    "color": { /* ... */ },
    "spacing": { /* ... */ },
    "typography": { /* ... */ },
    "shadow": { /* ... */ },
    "borderRadius": { /* ... */ }
  }
})
// → { success: true }

// Step 3 — define components
set_design_system({
  projectId: "my-site",
  designSystem: {
    "$schema": "https://design-tokens.org/schema.json",
    "_meta": { /* ... */ },
    "components": [ /* 20+ components */ ]
  }
})

// Step 4 — get resolved values, generate HTML
get_design_system({ projectId: "my-site" })
// tokenBindings: { backgroundColor: "#4F46E5" } ← resolved
Ordering constraint: set_tokens must be called before set_design_system. add_component and remove_component require an existing design system. Calls made out of order return a structured error.

Incremental updates

After the initial setup, use add_component to extend the design system without rewriting it:

add_component({
  projectId: "my-site",
  component: {
    id: "pricing-card",
    displayName: "Pricing Card",
    category: "data-display",
    description: "Pricing tier card with plan name and CTA.",
    isContainer: false,
    allowedChildren: "none",
    allowedParents: "*",
    props: {
      plan:  { type: "string", required: true },
      price: { type: "string", required: true }
    },
    slots: [],
    tokenBindings: {
      backgroundColor: "{color.background.surface}",
      padding:          "{spacing.lg}"
    },
    requiredBindingKeys: ["padding"]
  }
})

Environment variables

VariableDefaultDescription
INKBOOM_DEBUG — Set to any value to log raw token/component output on validation failure. Useful when debugging why a set_tokens call is rejected.

Development

terminal 
❯ pnpm test        # run tests with vitest
❯ pnpm typecheck   # tsc --noEmit
❯ pnpm format      # prettier --write .
❯ pnpm build       # tsc (emits to dist/)

Test files live in tests/, mirroring the src/ structure. Never place test files next to source files.
